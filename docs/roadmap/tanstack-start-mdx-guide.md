# Building an MDX-Powered Site with TanStack Start

This guide covers two production-ready ways to build a content site (blog, docs, articles) with [TanStack Start](https://tanstack.com/start) where every page or post is an `.mdx` file — Markdown with embedded React components.

**Approach A — direct compilation with `@mdx-js/rollup`.** Each `.mdx` file becomes an importable React component inside Vite's module graph. Minimal moving parts, per-post code splitting, and you can `import` components right inside a post.

**Approach B — Content Collections.** A dedicated content layer that parses, _validates_, and compiles your MDX into a generated, fully typed module. Frontmatter is checked against a Zod schema at build time, and TypeScript knows the exact shape of every post.

Both share the same TanStack Start skeleton, so the guide starts there. A comparison and gotchas section is at the end.

> TanStack Start evolves quickly (it was still pre-1.0 in 2026). The patterns here are stable, but cross-check exact plugin options against the [official docs](https://tanstack.com/start/latest) for your installed version.

---

## 1. Shared project setup

Scaffold a project with the official CLI (requires a recent Node LTS):

```bash
npm create @tanstack/start@latest my-site
cd my-site
```

The parts of the resulting structure that matter for this guide, plus the directories we'll add:

```text
my-site/
├── content/
│   └── posts/                  # your .mdx files live here, NOT in src/routes
│       ├── hello-world.mdx
│       └── second-post.mdx
├── src/
│   ├── components/
│   │   ├── Callout.tsx
│   │   └── mdx-components.tsx  # shared component map for MDX rendering
│   ├── routes/
│   │   ├── __root.tsx
│   │   ├── index.tsx
│   │   └── blog/
│   │       ├── index.tsx       # post listing
│   │       └── $slug.tsx       # individual post
│   ├── router.tsx
│   └── routeTree.gen.ts        # generated by the router plugin
├── content-collections.ts      # Approach B only
├── vite.config.ts
└── tsconfig.json
```

Content lives in `content/` at the project root rather than inside `src/routes`, because the TanStack Router plugin scans `src/routes` and treats files there as route definitions. Keeping prose out of that directory avoids codegen surprises and keeps the mental model clean: routes are code, content is data.

A shared custom component both approaches will use, `src/components/Callout.tsx`:

```tsx
import type { ReactNode } from 'react';

export function Callout({ type = 'info', children }: { type?: 'info' | 'warn'; children: ReactNode }) {
  return <aside className={type === 'warn' ? 'my-4 rounded border-l-4 border-amber-500 bg-amber-50 p-4' : 'my-4 rounded border-l-4 border-blue-500 bg-blue-50 p-4'}>{children}</aside>;
}
```

## 2. Approach A: direct MDX compilation with `@mdx-js/rollup`

### 2.1 Install

```bash
npm i @mdx-js/rollup remark-gfm remark-frontmatter remark-mdx-frontmatter
npm i -D @types/mdx
```

`@mdx-js/rollup` is the official MDX plugin for Rollup and Vite. `remark-gfm` enables tables, strikethrough, task lists, and autolinks. `remark-frontmatter` teaches the parser to recognize the YAML block, and `remark-mdx-frontmatter` turns that block into a named `frontmatter` export on every compiled module — which is what makes globbing metadata possible later. `@types/mdx` supplies the `mdx/types` module (for `MDXComponents`) and basic typings for `.mdx` imports.

### 2.2 Configure Vite

```ts
// vite.config.ts
import { defineConfig } from 'vite';
import { tanstackStart } from '@tanstack/react-start/plugin/vite';
import viteReact from '@vitejs/plugin-react';
import mdx from '@mdx-js/rollup';
import remarkGfm from 'remark-gfm';
import remarkFrontmatter from 'remark-frontmatter';
import remarkMdxFrontmatter from 'remark-mdx-frontmatter';

export default defineConfig({
  plugins: [
    // enforce: 'pre' makes MDX compile to plain JS before any other plugin
    // sees the module, so Start and React treat posts like ordinary files
    {
      enforce: 'pre',
      ...mdx({
        remarkPlugins: [remarkGfm, remarkFrontmatter, remarkMdxFrontmatter],
      }),
    },
    tanstackStart(),
    // the Start docs require React's plugin to come after Start's;
    // the include pattern extends Fast Refresh to .mdx files
    viteReact({ include: /\.(js|jsx|ts|tsx|md|mdx)$/ }),
  ],
});
```

Ordering is the part people get wrong. The MDX plugin must run in the `pre` phase, and `viteReact` must come after `tanstackStart`. If MDX runs too late you'll see errors like _"The JSX syntax extension is not currently enabled"_ coming from esbuild.

### 2.3 Author content

`content/posts/hello-world.mdx`:

```mdx
---
title: 'Hello World'
date: '2026-06-12'
description: 'First post on the new site'
---

import { BigChart } from '../../src/components/BigChart';

# {frontmatter.title}

Regular **markdown** prose, GFM tables, `inline code` — all of it works.

<Callout type='info'>A custom React element used directly in markdown. `Callout` is injected via the shared components map, so posts don't need to import it.</Callout>

<BigChart data={[3, 1, 4, 1, 5]} />
```

Two things to notice. First, `{frontmatter.title}` works because `remark-mdx-frontmatter` puts the parsed frontmatter in scope of the compiled module. Second, there are two ways to use a component: import it explicitly inside the file (good for one-off, heavy components like `BigChart`), or rely on the components map passed at render time (good for the shared vocabulary every post uses — callouts, figures, styled headings).

### 2.4 The listing route

`src/routes/blog/index.tsx`:

```tsx
import { createFileRoute, Link } from '@tanstack/react-router';

export interface Frontmatter {
  title: string;
  date: string;
  description?: string;
}

// `import: 'frontmatter'` pulls only the metadata export of each module,
// so the index page doesn't bundle every post body
export const postMeta = import.meta.glob<Frontmatter>('/content/posts/*.mdx', { eager: true, import: 'frontmatter' });

const posts = Object.entries(postMeta)
  .map(([path, fm]) => ({
    slug: path
      .split('/')
      .pop()!
      .replace(/\.mdx$/, ''),
    ...fm,
  }))
  .sort((a, b) => b.date.localeCompare(a.date));

export const Route = createFileRoute('/blog/')({
  component: BlogIndex,
});

function BlogIndex() {
  return (
    <main className='mx-auto max-w-2xl px-4 py-8'>
      <h1 className='text-3xl font-bold'>Blog</h1>
      <ul className='mt-8 space-y-6'>
        {posts.map(post => (
          <li key={post.slug}>
            <Link to='/blog/$slug' params={{ slug: post.slug }} className='text-xl font-semibold underline'>
              {post.title}
            </Link>
            <p className='text-sm text-gray-500'>{post.date}</p>
            {post.description && <p>{post.description}</p>}
          </li>
        ))}
      </ul>
    </main>
  );
}
```

`import.meta.glob` with a leading `/` resolves from the project root, which is how the route reaches `content/` outside of `src/`. The slug is derived from the filename, so `content/posts/hello-world.mdx` is served at `/blog/hello-world`.

### 2.5 The shared component map

`src/components/mdx-components.tsx`:

```tsx
import type { MDXComponents } from 'mdx/types';
import { Callout } from './Callout';

export const mdxComponents: MDXComponents = {
  h2: props => <h2 className='mt-10 text-2xl font-semibold' {...props} />,
  a: props => <a className='text-blue-600 underline' {...props} />,
  pre: props => <pre className='overflow-x-auto rounded bg-gray-900 p-4 text-gray-100' {...props} />,
  Callout,
};
```

Lowercase keys override the HTML elements markdown produces (`## heading` renders through your `h2`). Capitalized keys become components usable in any post without an import statement.

### 2.6 The post route

`src/routes/blog/$slug.tsx`:

```tsx
import { createFileRoute, notFound } from '@tanstack/react-router';
import { lazy, Suspense } from 'react';
import type { ComponentType, LazyExoticComponent } from 'react';
import type { MDXComponents } from 'mdx/types';
import { mdxComponents } from '../../components/mdx-components';
import { postMeta } from './index';

type PostComponent = ComponentType<{ components?: MDXComponents }>;

// no `eager` here: each post becomes its own async chunk
const modules = import.meta.glob<{ default: PostComponent }>('/content/posts/*.mdx');

const pathFor = (slug: string) => `/content/posts/${slug}.mdx`;

// lazy() must not be re-created on every render, otherwise React unmounts
// the subtree and Suspense re-triggers on each state change
const cache = new Map<string, LazyExoticComponent<PostComponent>>();

function getPostComponent(slug: string) {
  const key = pathFor(slug);
  let component = cache.get(key);
  if (!component) {
    component = lazy(modules[key]);
    cache.set(key, component);
  }
  return component;
}

export const Route = createFileRoute('/blog/$slug')({
  loader: ({ params }) => {
    if (!modules[pathFor(params.slug)]) throw notFound();
  },
  head: ({ params }) => {
    const fm = postMeta[pathFor(params.slug)];
    return {
      meta: [{ title: fm ? `${fm.title} — My Site` : 'My Site' }, { name: 'description', content: fm?.description ?? '' }],
    };
  },
  component: PostPage,
});

function PostPage() {
  const { slug } = Route.useParams();
  const Post = getPostComponent(slug);

  return (
    <article className='prose mx-auto max-w-2xl px-4 py-8'>
      <Suspense fallback={null}>
        <Post components={mdxComponents} />
      </Suspense>
    </article>
  );
}
```

How this behaves:

The loader runs on the server during SSR (and on the client during navigation). Throwing `notFound()` there gives unknown slugs a proper 404 instead of a runtime crash. The `head` option reuses the eager frontmatter glob from the listing route, so every post gets a correct `<title>` and meta description server-side — important for SEO and link previews.

Rendering uses the lazy glob, which means each post is split into its own chunk: visiting one post downloads one post. React's streaming server renderer resolves `lazy` components during SSR, so the full article HTML is in the initial response, not filled in after hydration.

If your site has only a handful of posts, you can skip the lazy machinery entirely: glob with `{ eager: true }` and render `modules[pathFor(slug)].default` synchronously. Simpler code, one bigger chunk.

## 3. Approach B: Content Collections (type-safe frontmatter)

[Content Collections](https://www.content-collections.dev) sits between your files and your app. It watches `content/`, parses frontmatter, validates it against a schema you define, optionally transforms each document (this is where MDX compilation happens), and emits a generated module you import as `content-collections`. The payoff: invalid frontmatter fails the build with a precise error, and `allPosts` is typed end-to-end without you writing a single interface.

### 3.1 Install and wire up

```bash
npm i @content-collections/core @content-collections/vite @content-collections/mdx zod
```

Map the generated module in `tsconfig.json` so TypeScript can resolve it:

```json
{
  "compilerOptions": {
    "paths": {
      "content-collections": ["./.content-collections/generated"]
    }
  }
}
```

Ignore the generated output:

```gitignore
.content-collections
```

Register the Vite plugin — it must be **first** in the list so the generated module exists before Start or React try to resolve imports of it (the plugin also handles the `content-collections` alias on the Vite side, mirroring the tsconfig path):

```ts
// vite.config.ts
import { defineConfig } from 'vite';
import contentCollections from '@content-collections/vite';
import { tanstackStart } from '@tanstack/react-start/plugin/vite';
import viteReact from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [contentCollections(), tanstackStart(), viteReact()],
});
```

Note that `@mdx-js/rollup` is gone from the config: in this approach MDX is compiled by the content layer, not by Vite's module graph.

### 3.2 Define the collection

`content-collections.ts` at the project root:

```ts
import { defineCollection, defineConfig } from '@content-collections/core';
import { compileMDX } from '@content-collections/mdx';
import remarkGfm from 'remark-gfm';
import { z } from 'zod';

const posts = defineCollection({
  name: 'posts',
  directory: 'content/posts',
  include: '**/*.mdx',
  schema: z.object({
    title: z.string(),
    date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/, 'expected YYYY-MM-DD'),
    description: z.string().optional(),
    tags: z.array(z.string()).default([]),
    draft: z.boolean().default(false),
    // the markdown body arrives on `content`; Zod strips undeclared keys,
    // so it must be listed or the transform loses access to it
    content: z.string(),
  }),
  transform: async (document, context) => {
    const mdx = await compileMDX(context, document, {
      remarkPlugins: [remarkGfm],
    });
    return {
      ...document,
      slug: document._meta.path,
      mdx,
    };
  },
});

export default defineConfig({
  collections: [posts],
});
```

What each piece does:

**`schema`** is a plain Zod object (current Content Collections versions accept any Standard Schema validator directly — Zod, Valibot, ArkType). Every `.mdx` file's frontmatter is validated against it when the dev server starts and on every file change. A missing `title` or a malformed `date` stops the build and points at the offending file, instead of rendering `undefined` somewhere in production.

**`transform`** runs per document. `compileMDX` compiles the markdown body to executable component code (a string), which is stored alongside the metadata. Adding `slug` here means every consumer gets it for free. Crucially, TypeScript _infers_ the post type from schema + transform — whatever you return is what `allPosts[number]` looks like.

**`_meta`** is attached to every document automatically and carries file information: `path` (file path without extension, ideal as a slug), `fileName`, `directory`, `filePath`, `extension`.

Frontmatter for this approach (note the quoted date — see gotchas):

```mdx
---
title: 'Hello World'
date: '2026-06-12'
description: 'First post on the new site'
tags: ['meta']
---

# Hello

<Callout type='info'>Custom components still work — see below.</Callout>
```

### 3.3 The listing route

`src/routes/blog/index.tsx`:

```tsx
import { createFileRoute, Link } from '@tanstack/react-router';
import { allPosts } from 'content-collections';

const posts = allPosts.filter(post => !post.draft).sort((a, b) => b.date.localeCompare(a.date));

export const Route = createFileRoute('/blog/')({
  component: BlogIndex,
});

function BlogIndex() {
  return (
    <main className='mx-auto max-w-2xl px-4 py-8'>
      <h1 className='text-3xl font-bold'>Blog</h1>
      <ul className='mt-8 space-y-6'>
        {posts.map(post => (
          <li key={post.slug}>
            <Link to='/blog/$slug' params={{ slug: post.slug }} className='text-xl font-semibold underline'>
              {post.title}
            </Link>
            <p className='text-sm text-gray-500'>{post.date}</p>
            {post.description && <p>{post.description}</p>}
          </li>
        ))}
      </ul>
    </main>
  );
}
```

No glob, no path string parsing, no hand-written `Frontmatter` interface. `post.title` autocompletes; `post.titel` is a compile error. Rename `description` to `summary` in the schema and every route that still reads `description` turns red immediately.

### 3.4 The post route

`src/routes/blog/$slug.tsx`:

```tsx
import { createFileRoute, notFound } from '@tanstack/react-router';
import { allPosts } from 'content-collections';
import { MDXContent } from '@content-collections/mdx/react';
import { mdxComponents } from '../../components/mdx-components';

export const Route = createFileRoute('/blog/$slug')({
  loader: ({ params }) => {
    const post = allPosts.find(p => p.slug === params.slug && !p.draft);
    if (!post) throw notFound();
    return { post };
  },
  head: ({ loaderData }) => ({
    meta: [{ title: loaderData ? `${loaderData.post.title} — My Site` : 'My Site' }, { name: 'description', content: loaderData?.post.description ?? '' }],
  }),
  component: PostPage,
});

function PostPage() {
  const { post } = Route.useLoaderData();

  return (
    <article className='prose mx-auto max-w-2xl px-4 py-8'>
      <h1>{post.title}</h1>
      <MDXContent code={post.mdx} components={mdxComponents} />
    </article>
  );
}
```

The whole post — metadata and compiled code string — is serializable, so it flows through the loader and TanStack Start's SSR hydration without ceremony. `MDXContent` evaluates the compiled code at render time and mounts it with your components map. No `lazy`, no `Suspense`, no module cache.

### 3.5 Two constraints to know about

**No `import` statements inside `.mdx` files.** The content layer compiles each file in isolation; it doesn't bundle imports the way Vite does in Approach A. Every custom component a post uses must be registered in the components map passed to `MDXContent`. For a controlled design system of content components this is arguably a feature — posts can't reach into arbitrary app code. Related: `{frontmatter.title}` expressions aren't in scope either; render metadata from the route (as the `<h1>` above does).

**Runtime evaluation.** `MDXContent` executes the compiled code string via the `Function` constructor. Under a strict Content Security Policy that forbids `'unsafe-eval'`, this fails in the browser. If you run a strict CSP, either prerender posts to static HTML on the server, or use Approach A, where MDX is compiled to ordinary modules at build time and nothing is evaluated at runtime.

## 4. Choosing between the approaches

| Concern                                            | A: `@mdx-js/rollup`                  | B: Content Collections                      |
| -------------------------------------------------- | ------------------------------------ | ------------------------------------------- |
| Frontmatter validation                             | None — typos surface at runtime      | Zod schema, build fails with precise errors |
| Post types                                         | Hand-written interface, trusted      | Inferred from schema + transform            |
| Components in posts                                | `import` in-file _or_ components map | Components map only                         |
| Frontmatter in post body                           | `{frontmatter.title}` works          | Render from route instead                   |
| Code splitting                                     | One chunk per post via lazy glob     | Compiled code ships in collection data      |
| Strict CSP (no `unsafe-eval`)                      | Fine — build-time compilation        | Needs prerendering or relaxed CSP           |
| Extra machinery                                    | None beyond Vite plugins             | Config file, generated module, watcher      |
| Computed fields (slug, reading time, sorting keys) | DIY in route code                    | One place: `transform`                      |

A reasonable default: start with Approach A for a small personal site where you want maximum flexibility inside posts; reach for Content Collections once content volume grows, multiple people write posts, or frontmatter consistency starts to matter — schema validation is exactly the guardrail that prevents "the deploy is broken because someone wrote `data:` instead of `date:`".

The two aren't religiously exclusive, but pick one MDX pipeline per project; running both compilers over the same files buys confusion, not capability.

## 5. Gotchas and production notes

**YAML eats your dates.** Unquoted `date: 2026-06-12` is parsed by YAML into a `Date` object before any schema sees it, which then fails `z.string()` or serializes inconsistently. Quote dates in frontmatter (`date: "2026-06-12"`) and validate the string format in the schema, as done above.

**Plugin order is load-bearing.** Recap: in Approach A the MDX plugin carries `enforce: 'pre'` and `viteReact` comes after `tanstackStart`; in Approach B `contentCollections()` is the first plugin. Most cryptic startup errors in these setups trace back to ordering.

**Keep content out of `src/routes`.** Worth repeating: the router plugin owns that directory. `.mdx` files there will at best be ignored and at worst confuse route generation.

**The `lazy()` cache in Approach A is not optional.** Calling `lazy(modules[key])` inline during render creates a new component identity each time, so any re-render unmounts the article and flashes the Suspense fallback. The module-level `Map` keyed by path is the standard fix.

**Static output.** Both approaches pair well with TanStack Start's prerendering for a fully static site (no server at all). Recent versions expose it on the Start plugin:

```ts
tanstackStart({
  prerender: {
    enabled: true,
    crawlLinks: true,
  },
});
```

With `crawlLinks`, the prerenderer starts at the root, follows your `<Link>`s to every post, and writes static HTML for each. Verify the exact option names against the docs for your installed version, since this API has been moving.

**Styling prose.** Markdown output is bare HTML elements. Either override elements in the components map (fine-grained control) or wrap the article in Tailwind's `@tailwindcss/typography` `prose` class and only override what the plugin doesn't cover.

**RSS / sitemaps.** Both approaches give you everything needed at build time — Approach A via the eager frontmatter glob, Approach B via `allPosts` — so a server route (e.g. `src/routes/rss[.]xml.ts`) can assemble a feed from the same data the listing page uses.
